# ADR-007: Cloudflare R2 for Image Storage
**Date:** 2026-05-30 | **Status:** Accepted | **Supersedes:** Local filesystem (OQ-002)

## Problem
Blog thumbnail and inline body images need to be stored somewhere. Local disk storage creates operational burden (disk quotas, backups, scaling).

## Decision
Use Cloudflare R2 (S3-compatible object storage) for all image uploads.

## Upload Flow
```
Client → POST /uploads/image (multipart) → Go API
  → validate type (JPEG/PNG/WEBP) + size (≤ 5MB)
  → upload to R2 bucket via aws-sdk-go-v2 (R2 endpoint)
  → return public R2 URL
  → store R2 URL + object key in images table
```

## Environment Variables Required
```
R2_ACCOUNT_ID=
R2_ACCESS_KEY_ID=
R2_SECRET_ACCESS_KEY=
R2_BUCKET_NAME=
R2_PUBLIC_URL=       # e.g. https://pub-xxx.r2.dev or custom domain
```

## Reasons
- Owner does not want images on local disk
- R2 has no egress fees (cost-efficient for image serving)
- S3-compatible API — standard SDK works
- Cloudflare CDN serves images globally with low latency
- Easy migration path — already on object storage from day one

## Consequences
- Requires R2 bucket + API credentials before running the app
- `images` table stores `r2_key` (object key) + `url` (public URL) instead of `filepath`
- Upload service must be configured with R2 credentials via env vars
- Local dev can use R2 dev bucket or mock (no local fallback needed)
